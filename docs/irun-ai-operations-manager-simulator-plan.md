# iRun AI 运维经理模拟器改造方案

更新日期：2026-06-29

## 1. 改造结论

本轮改造的方向不是把当前 iRun 继续做成「更炫的运维大屏」，而是升级为「iRun AI 运维经理模拟器」。

核心变化：

- 过去的产品最小单元是「一块给人操作的页面」。
- 新形态的产品最小单元是「一个能自己闭环的任务」。
- 过去强调告警、工单、调度等工程效率。
- 新命题强调如何经营一支可复制、可托管、持续进化的数字劳动力。

因此，当前项目中间的大屏背景、背景切换、电站视觉资产继续保留，作为演示舞台；需要升级的是外围布局、叙事结构、交互决策和组织管理表达。

## 2. 产品定位

### 2.1 一句话定位

iRun AI 是一套面向新能源运维的数字劳动力管理界面，让用户像管理一支团队一样雇佣、编排、授权、监督和复盘 AI 运维员工。

### 2.2 展示主张

你不是在买一套软件，而是在雇佣一支永远在岗、可批量复制、能持续学习的数字运维团队。

### 2.3 体验目标

- 让观众从「看系统」切换到「指挥团队」。
- 让 AI 从单点功能变成有组织分工的数字员工。
- 让运维闭环从页面操作变成任务自治。
- 让关键交互集中在授权、仲裁、托管和复盘。

## 3. 保留与改造边界

### 3.1 保留

- 中间大屏背景图、视频背景和背景切换逻辑。
- 当前总览地图、电站 pin、单站现场背景。
- 现有 10 个智能体基础数据。
- 当前多 Agent 连线、机器人行走、UAV 巡检等舞台动效能力。

### 3.2 改造

- 顶部 KPI 从电站监控指标升级为数字劳动力经营指标。
- 左侧事件流升级为 S1-S10 场景导演轨。
- 右侧对话调度升级为运维经理决策台。
- 底部 Token 面板升级为数字团队组织面板。
- 现有自动场景 A/B 升级为完整 10 幕叙事。
- 指挥模式从「聊天派遣」升级为「关键判断与授权」。

## 4. 信息架构建议

### 4.1 中央舞台：AI 运维现场

中央区域继续承载电站、地图、巡检、事件、连线和机器人动作。

建议减少长期遮挡中央区域的 UI 卡片，把中央舞台交给以下内容：

- 多电站在岗状态。
- 事件发生位置和影响范围。
- Agent 行动轨迹。
- Agent 协同连线。
- 任务闭环进度。
- 托管模式下的时间加速反馈。

### 4.2 左侧：场景导演轨

替代当前普通事件流，成为展会讲解员的主控节奏。

内容结构：

- 当前场景编号：S1 至 S10。
- 每一幕一句话。
- 场景状态：装置、工作台、混合仪式。
- 交互类型：自动循环、讲解员触发、上手交互、自动收尾。
- 当前演出进度。
- 讲解员按钮：开始、下一幕、触发告警、进入经理模式、启动托管。

### 4.3 右侧：运维经理决策台

替代当前对话调度面板，重点不是聊天，而是做管理判断。

建议模块：

- 当前任务：事件、影响范围、优先级。
- AI 团队建议：推荐动作、理由、置信度。
- 证据链：告警、图像、工单、天气、安全规则。
- 分歧仲裁：不同 Agent 的判断差异。
- 经理操作：批准、复核、转人工、交给 AI 托管。
- 后果预览：预计恢复时间、风险变化、成本影响。

### 4.4 底部：数字团队组织面板

替代或重构当前 Token 面板和 Agent 竖栏，把智能体表达成一支团队。

组织分层：

- 工具型：问数、光伏助手。
- 专家型：预警、诊断、安全、巡检。
- 流程型：告警、工单、排程。
- 管理型：运营。

每个数字员工建议展示：

- 在岗状态。
- 当前任务。
- 负载。
- 置信度。
- 可托管等级。
- 最近一次贡献。
- 是否参与当前闭环。

### 4.5 顶部：公司层经营指标

将当前电站 KPI 升级为数字劳动力经营结果。

建议指标：

- 同时在岗电站数。
- 今日自动闭环任务数。
- 人工介入次数。
- 累计节省工时。
- 托管可信度。
- 经验回流数量。
- 跨站复用策略数。

## 5. S1-S10 场景规划

| 场景 | 一句话 | 态 | 交互类型 | 页面表达建议 |
| --- | --- | --- | --- | --- |

| S4 事件触发 | AI 团队开始响应事件 | 工作台 | 讲解员可触发告警 | 讲解员触发告警，现场出现事件波纹、任务卡和响应倒计时 |
| S5 多 Agent 协同 | AI 不是单点功能，是团队协同 | 工作台 | 自动演出 | 展示告警、诊断、工单、排程、安全、运营之间的接力 |

| S7 当一天经理 | AI 备好上下文，你只做关键判断 | 工作台 | 上手交互 | 用户选择批准、复核或转人工，后续由 AI 自动执行 |
| S8 闭环汇报 | 一天的运维效率跃升，可量化 | 工作台 | 自动呈现 | 自动生成闭环汇报，展示节省工时、风险下降、经验沉淀 |
| S9 托管电站 | 时间加速，AI 自主跑完周期 | 混合仪式 | 上手交互 | 用户调节托管等级，系统时间加速并跑完整运维周期 |
| S10 拉回东盟总部 | 成果汇入公司层，经验回流 | 装置 | 自动收尾 | 回到总览，公司层显示经验回流、策略复制和区域效率提升 |

## 6. 两个核心上手交互

### 6.1 S7 当一天经理

目标：让用户体验「AI 已经准备好上下文，人只做关键判断」。

建议交互：

1. 事件发生后，右侧出现 AI 推荐方案。
2. 系统展示证据链和多 Agent 共识。
3. 用户在三个决策卡中选择：
   - 批准 AI 推荐方案。
   - 要求安全智能体复核。
   - 转人工处理。
4. 选择后，中间舞台自动演出后续闭环。
5. 底部团队面板更新每个 Agent 的贡献。

建议文案：

- 「AI 已完成研判，你只需要决定是否授权执行。」
- 「推荐派工：2 小时内到场，风险可下降 67%。」
- 「安全智能体建议增加绝缘复测，预计多耗时 18 分钟。」

### 6.2 S9 托管电站

目标：让用户体验「把一个电站交给 AI 跑完整周期」。

建议交互：

托管滑杆：

- L1 观察：AI 只提示，不执行。
- L2 建议：AI 生成方案，等待经理批准。
- L3 半自动：低风险任务自动执行，高风险任务等待批准。
- L4 全托管：AI 自主闭环，经理只接收异常和日报。

演出反馈：

- 时间加速条从 08:00 推进到 18:00。
- 巡检、告警、工单、复盘按时间线自动出现。
- 中央舞台展示任务闭环动画。
- 顶部经营指标实时变化。
- 最后生成一天托管结果。

## 8. 状态机建议

建议引入统一的模拟器状态，而不是让每个 UI 模块各自判断。

核心状态：

```js
{
  sceneId: 'S1',
  sceneMode: 'installation' | 'workbench' | 'ritual',
  playState: 'idle' | 'playing' | 'paused' | 'waiting-user',
  focusPlantId: null,
  selectedAgentId: null,
  currentIncidentId: null,
  managerDecision: null,
  autonomyLevel: 2,
  elapsedSimTime: 0
}
```

这样可以让：

- 左侧导演轨控制全局进度。
- 右侧经理台根据场景显示不同决策。
- 中央舞台根据状态播放动作。
- 底部团队面板展示当前参与者。

## 10. 视觉与动效原则

视觉方向：

- 保持暗色科技风，但减少纯装饰发光。
- 用组织、证据、授权、状态表达高级感。
- 中央舞台要清爽，外围 UI 承担解释和控制。
- 卡片不宜过多，重点信息用状态条、任务流、关系线表达。

动效方向：

- 事件触发：位置涟漪、告警点亮、任务卡生成。
- 团队协同：Agent 接力、连线流动、证据汇聚。
- 仲裁判断：分歧并列、置信度收敛、结论落锤。
- 经理授权：决策卡确认、任务进入执行队列。
- 托管电站：时间加速、周期推进、汇报生成。

注意：

- 长时间自动循环动效要可暂停。
- 交互动效控制在 150-300ms。
- 对展会大屏而言，文本要少而准，关键标题要可远距离识别。

## 12. 开发拆解建议

第一批可以先做低风险结构改造：

- 新增 `SIMULATOR_SCENES` 数据。
- 新增 `simulatorState` 全局状态。
- 新建 `ScenarioDirectorRail`。
- 新建 `ManagerDecisionConsole`。
- 新建 `DigitalTeamOrgPanel`。
- 保持当前背景、地图和单站舞台不动。

第二批再替换现有区域：

- 左侧由 `EventStream` 切到 `ScenarioDirectorRail`。
- 右侧由 `DispatchPanel` 切到 `ManagerDecisionConsole`。
- 底部由 `AgentTokenPanel` 切到 `DigitalTeamOrgPanel`。
- 旧组件保留一版开关，方便对照和回滚。

第三批补完整交互：

- S7 决策卡。
- S9 托管等级滑杆。
- 场景自动播放和暂停。
- 讲解员下一幕控制。
- 闭环汇报生成。

## 13. 后续待确认问题

- 展会讲解时长是 3 分钟、5 分钟还是 8 分钟？
- S1 的 90 秒自动循环是否需要常驻，还是只作为首页 idle 状态？
- S2 人才市场是否需要真实定价、雇佣流程、合同感，还是只做概念演示？
- S7 的经理决策是否固定三选一，还是允许自由输入？
- S9 的全托管是否需要表现风险边界和人工接管机制？
- 是否需要中英文双语完整覆盖？
- GitHub Pages 展示版是否继续保持公开仓库？
