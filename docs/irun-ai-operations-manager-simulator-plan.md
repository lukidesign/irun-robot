# iRun AI 运维经理模拟器改造方案

更新日期：2026-06-29

## 1. 改造结论

本轮改造的方向不是把当前 iRun 继续做成「更炫的运维大屏」，而是升级为「iRun AI 运维经理模拟器」。

核心变化：

- 过去的产品最小单元是「一块给人操作的页面」。
- 新形态的产品最小单元是「一个能自己闭环的任务」。
- 过去强调告警、工单、调度等工程效率。
- 新命题强调如何经营一支可复制、可托管、持续进化的数字劳动力。

因此，当前项目中间的大屏背景、背景切换、电站视觉资产继续保留，作为演示舞台；需要升级的是外围布局、叙事结构、交互决策和组织管理表达。

硬性约束：现有总览大屏不是被替换的首页，而是新模拟器的核心舞台。当前总览背景、白天/黑夜主题切换、地图模式、电站 pin、视频/图片背景切换都应继续保留；新增的导演轨、经理决策台、数字团队面板和游戏化反馈只能围绕该舞台增强，不应重建一个独立 landing page 或长期遮挡中央舞台。

## 2. 产品定位

### 2.1 一句话定位

iRun AI 是一套面向新能源运维的数字劳动力管理界面，让用户像管理一支团队一样雇佣、编排、授权、监督和复盘 AI 运维员工。

### 2.2 展示主张

你不是在买一套软件，而是在雇佣一支永远在岗、可批量复制、能持续学习的数字运维团队。

### 2.3 体验目标

- 让观众从「看系统」切换到「指挥团队」。
- 让 AI 从单点功能变成有组织分工的数字员工。
- 让运维闭环从页面操作变成任务自治。
- 让关键交互集中在授权、仲裁、托管和复盘。
- 让观众获得轻量游戏式参与感：接任务、做选择、看反馈、完成闭环、获得可量化结果。

## 3. 保留与改造边界

### 3.1 保留

- 中间大屏背景图、视频背景和背景切换逻辑。
- 当前总览大屏背景和白天/黑夜主题切换。
- 当前 `map2` 总览模式、图1/图2/展示/漫游等子模式。
- 当前总览地图、电站 pin、单站现场背景。
- 现有 10 个智能体基础数据。
- 当前多 Agent 连线、机器人行走、UAV 巡检等舞台动效能力。

### 3.2 改造

- 顶部 KPI 从电站监控指标升级为数字劳动力经营指标。
- 左侧事件流升级为 S1-S10 场景导演轨。
- 右侧对话调度升级为运维经理决策台。
- 底部 Token 面板升级为数字团队组织面板。
- 现有自动场景 A/B 升级为完整 10 幕叙事。
- 指挥模式从「聊天派遣」升级为「关键判断与授权」。

### 3.3 游戏互动边界

游戏感不是把系统改成传统游戏，而是增加「任务感、选择感、反馈感和完成感」。

应增加：

- 任务目标：每一幕都有明确目标，例如触发告警、完成授权、降低风险、托管一天。
- 决策反馈：用户选择后立即看到风险、效率、信任度、成本等指标变化。
- 进度推进：用场景进度、任务倒计时、闭环进度条表达推进感。
- 轻量奖励：完成关键动作后出现勋章、评级或经营结果提升。
- 可重玩分支：S7 的不同决策带来不同后果预览和汇报结果。

应避免：

- 把真实运维风险娱乐化。
- 用过多浮夸特效遮挡总览背景和中央舞台。
- 引入复杂键盘操作或高学习成本玩法。
- 为了游戏感牺牲展会讲解员的可控节奏。

## 4. 信息架构建议

### 4.1 中央舞台：AI 运维现场

中央区域继续承载电站、地图、巡检、事件、连线和机器人动作。

建议减少长期遮挡中央区域的 UI 卡片，把中央舞台交给以下内容：

- 多电站在岗状态。
- 事件发生位置和影响范围。
- Agent 行动轨迹。
- Agent 协同连线。
- 任务闭环进度。
- 托管模式下的时间加速反馈。

### 4.2 左侧：场景导演轨

替代当前普通事件流，成为展会讲解员的主控节奏。

内容结构：

- 当前场景编号：S1 至 S10。
- 每一幕一句话。
- 场景状态：装置、工作台、混合仪式。
- 交互类型：自动循环、讲解员触发、上手交互、自动收尾。
- 当前演出进度。
- 讲解员按钮：开始、下一幕、触发告警、进入经理模式、启动托管。

### 4.3 右侧：运维经理决策台

替代当前对话调度面板，重点不是聊天，而是做管理判断。

建议模块：

- 当前任务：事件、影响范围、优先级。
- AI 团队建议：推荐动作、理由、置信度。
- 证据链：告警、图像、工单、天气、安全规则。
- 分歧仲裁：不同 Agent 的判断差异。
- 经理操作：批准、复核、转人工、交给 AI 托管。
- 后果预览：预计恢复时间、风险变化、成本影响。

### 4.4 底部：数字团队组织面板

替代或重构当前 Token 面板和 Agent 竖栏，把智能体表达成一支团队。

组织分层：

- 工具型：问数、光伏助手。
- 专家型：预警、诊断、安全、巡检。
- 流程型：告警、工单、排程。
- 管理型：运营。

每个数字员工建议展示：

- 在岗状态。
- 当前任务。
- 负载。
- 置信度。
- 可托管等级。
- 最近一次贡献。
- 是否参与当前闭环。

### 4.5 顶部：公司层经营指标

将当前电站 KPI 升级为数字劳动力经营结果。

建议指标：

- 同时在岗电站数。
- 今日自动闭环任务数。
- 人工介入次数。
- 累计节省工时。
- 托管可信度。
- 经验回流数量。
- 跨站复用策略数。

### 4.6 游戏化互动层：任务、反馈与奖励

游戏化互动层应跨越左右底三侧 UI，但不改变中央舞台的基础背景和昼夜切换。

建议元素：

- 任务目标条：显示当前场景目标、剩余步骤、是否等待用户决策。
- 决策后果预览：批准、复核、转人工分别展示效率、风险、成本和信任度影响。
- 经理评分：以「安全、效率、自治、收益」四类指标反馈选择结果。
- 团队士气/可信度：数字员工完成任务后提升托管可信度或跨站复用经验。
- 成就反馈：例如「首次授权」「零人工闭环」「安全复核完成」「跨站策略复用」。
- 复盘卡：场景结束后给出本轮表现，而不是只展示静态 KPI。

| S9 托管电站 | 时间加速，AI 自主跑完周期 | 混合仪式 | 上手交互 | 用户调节托管等级，系统时间加速并跑完整运维周期，给出托管得分 |

### 5.1 游戏互动节奏

每一幕都建议具备「目标 → 行动 → 反馈 → 进入下一幕」的节奏。

- S1/S10：偏装置感，强调大屏背景、昼夜切换和总部经营结果。
- S3-S5：偏任务推进，强调事件触发、团队接力和进度条。
- S6-S7：偏决策游戏，强调不同方案的后果预览。
- S8-S9：偏经营反馈，强调结果评分、托管等级和经验回流。

## 6. 核心上手交互与游戏化反馈

### 6.1 S7 当一天经理

目标：让用户体验「AI 已经准备好上下文，人只做关键判断」。

建议交互：

1. 事件发生后，右侧出现 AI 推荐方案。
2. 系统展示证据链和多 Agent 共识。
3. 用户在三个决策卡中选择：
   - 批准 AI 推荐方案。
   - 要求安全智能体复核。
   - 转人工处理。
4. 选择后，中间舞台自动演出后续闭环。
5. 底部团队面板更新每个 Agent 的贡献。

建议文案：

- 「AI 已完成研判，你只需要决定是否授权执行。」
- 「推荐派工：2 小时内到场，风险可下降 67%。」
- 「安全智能体建议增加绝缘复测，预计多耗时 18 分钟。」

### 6.2 S9 托管电站

目标：让用户体验「把一个电站交给 AI 跑完整周期」。

建议交互：

托管滑杆：

- L1 观察：AI 只提示，不执行。
- L2 建议：AI 生成方案，等待经理批准。
- L3 半自动：低风险任务自动执行，高风险任务等待批准。
- L4 全托管：AI 自主闭环，经理只接收异常和日报。

演出反馈：

- 时间加速条从 08:00 推进到 18:00。
- 巡检、告警、工单、复盘按时间线自动出现。
- 中央舞台展示任务闭环动画。
- 顶部经营指标实时变化。
- 最后生成一天托管结果。

### 6.3 游戏化反馈机制

目标：让用户每次操作都有可感知反馈，并知道自己的决策影响了什么。

建议机制：

1. 任务目标：每个场景进入时显示一个短目标，例如「在 20 秒内完成授权决策」。
2. 决策评分：用户选择后刷新四类分数：安全、效率、自治、收益。
3. 后果分支：批准 AI 推荐方案、要求安全复核、转人工处理会生成不同的闭环报告。
4. 团队贡献：参与本轮闭环的智能体获得贡献高亮，底部组织面板显示「本轮 MVP」。
5. 轻量成就：完成特定动作后出现短反馈，例如「零人工闭环」「安全复核通过」「经验已回流」。
6. 可重玩提示：闭环汇报中给出「如果选择安全复核，风险将再下降 8%，耗时增加 18 分钟」之类的对照。

游戏化反馈应服务于运维经理心智，重点是管理判断和组织调度，不做复杂操控型玩法。

## 8. 状态机建议

建议引入统一的模拟器状态，而不是让每个 UI 模块各自判断。

核心状态：

```js
{
  sceneId: 'S1',
  sceneMode: 'installation' | 'workbench' | 'ritual',
  playState: 'idle' | 'playing' | 'paused' | 'waiting-user',
  focusPlantId: null,
  selectedAgentId: null,
  currentIncidentId: null,
  managerDecision: null,
  autonomyLevel: 2,
  elapsedSimTime: 0,
  missionProgress: 0,
  missionGoal: 'approve-or-review',
  decisionScore: {
    safety: 82,
    efficiency: 76,
    autonomy: 68,
    business: 74
  },
  unlockedBadges: []
}
```

这样可以让：

- 左侧导演轨控制全局进度。
- 右侧经理台根据场景显示不同决策。
- 中央舞台根据状态播放动作。
- 底部团队面板展示当前参与者。
- 游戏化反馈层根据用户决策刷新进度、评分、成就和闭环结果。

## 10. 视觉与动效原则

视觉方向：

- 保持暗色科技风，但减少纯装饰发光。
- 保留当前总览大屏背景、白天/黑夜切换和舞台纵深感。
- 用组织、证据、授权、状态表达高级感。
- 中央舞台要清爽，外围 UI 承担解释和控制。
- 卡片不宜过多，重点信息用状态条、任务流、关系线表达。
- 游戏感优先使用任务条、进度、评分、短反馈和状态变化，不使用大面积游戏 UI 遮罩。

动效方向：

- 事件触发：位置涟漪、告警点亮、任务卡生成。
- 团队协同：Agent 接力、连线流动、证据汇聚。
- 仲裁判断：分歧并列、置信度收敛、结论落锤。
- 经理授权：决策卡确认、任务进入执行队列。
- 托管电站：时间加速、周期推进、汇报生成。
- 游戏反馈：目标完成、分数变化、成就短闪、团队 MVP 高亮。

注意：

- 长时间自动循环动效要可暂停。
- 交互动效控制在 150-300ms。
- 对展会大屏而言，文本要少而准，关键标题要可远距离识别。
- 所有游戏化反馈都要可跳过、可自动推进，保证讲解员控场。
